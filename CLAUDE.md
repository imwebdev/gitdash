# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## NORTH STAR — gitdash is for beginners

**The user of gitdash is a beginner.** Every interaction must be doable by clicking buttons in a browser. **No solution is acceptable that requires the end user to:**

- Open a terminal (PowerShell, WSL, Terminal, bash, anything)
- Type a shell command
- Edit a config file by hand
- Read a stack trace, error code, or technical jargon
- Know what `gh`, `git`, `systemd`, `wsl`, `npm`, or any CLI tool is

If a feature can fail in a way that requires the user to drop to a terminal to fix it, **that failure mode is a P0 bug** and must be fixed by adding a button in the UI that does the recovery automatically — not by writing a help-desk doc that says "run `gh auth login`".

This rule applies to:
- Install / uninstall (one-liner only; no manual cleanup)
- Auth (gh, git credentials) — must be a button + browser flow, not a terminal command
- Sync errors (push fails, pull fails, merge conflicts) — must surface in UI with a click-to-fix path
- Service management (start/stop/status) — must be a button, not `systemctl`
- Configuration changes — must be a settings page, not editing JSON

**When triaging any bug or implementing any feature, ask: "can a beginner who has never used a terminal recover from this on their own?" If the answer is no, add a button before shipping.**

## Workflow — GitHub-first (mandatory)

Inherited from `~/.claude/CLAUDE.md` and `~/CLAUDE.md`. **Every bug, feature, and task needs a GitHub issue before any code is written.** No exceptions, including when the user types a request directly into chat — first action is to create or find the issue.

Sequence for any change:

1. **Find or create the issue.** Check first: `gh issue list --state open` and `gh api repos/imwebdev/gitdash/milestones`. Reuse if it exists; create with `gh issue create` otherwise.
2. **Branch from main with the issue number:** `feature/<issue#>-<short-kebab-name>` (or `fix/<issue#>-...`, `chore/<issue#>-...`).
3. **Push and open a PR** that references the issue (`Closes #N` in the body).
4. **When the work is implemented:** add the `ready to test` label and comment on the issue with three things — what changed, the test scenario, and pass/fail criteria.
5. **Do not close the issue.** Issues stay open until the user (or the assigned tester) manually verifies the test scenario and closes it themselves.

Useful labels in this repo: `bug`, `enhancement`, `ready to test`. Add `phase-X` if working against a phased plan.

If the user asks you to "just do" something quick, the right answer is: create the issue first (it takes 10 seconds via `gh issue create --title "..." --body "..." --label "..."`), then work the issue. Skipping this is the violation, not the slowdown.

## Commands

```bash
npm run dev          # Next dev server on 127.0.0.1:7420
npm run build        # production build
npm start            # next start on 127.0.0.1:7420 (no CSRF token, dev-style)
npm run lint         # eslint via next lint
npm run typecheck    # tsc --noEmit (TypeScript strict)
npm run scan:once    # one-shot discovery + snapshot, no server (scripts/scan-once.ts)

./bin/gitdash start  # production launcher: generates CSRF, binds 0.0.0.0, opens browser
./bin/gitdash dev    # same plumbing, runs `next dev`
```

The `bin/gitdash` launcher is the supported way to run the tool. It generates `GITDASH_CSRF_TOKEN`, sets `GITDASH_PORT`, binds to `0.0.0.0` (so LAN access works), and the in-browser UI requires that token on all mutating requests. `npm start` skips this and is fine for local dev only.

There is no test suite. The README is a stale scaffold checklist; ignore it for status.

## Environment variables

- `GITDASH_PORT` (default `7420`)
- `GITDASH_BIND` (default `0.0.0.0` via launcher; `127.0.0.1` via `npm` scripts)
- `GITDASH_CSRF_TOKEN` (auto-generated by launcher; required by API routes for mutations)
- `GITDASH_DB` (default `~/.local/state/gitdash/gitdash.sqlite`)
- `GITDASH_EDITOR` (default `code` — VS Code; this codebase deliberately does NOT fall back to `$EDITOR`/`$VISUAL` because `vi` against a directory is a beginner trap)
- `GITDASH_TERMINAL` (default `x-terminal-emulator`)
- `XDG_CONFIG_HOME` — checked for `gitdash/config.json` to override `DiscoveryConfig` (see `lib/scan/discover.ts` for shape and defaults)

## Architecture

### Boot sequence

`lib/bootstrap.ts` is the single entry point called by every API route and the SSR page. It's idempotent (`bootstrapped` flag) and:
1. Loads optional `~/.config/gitdash/config.json`
2. Starts the **scheduler** (`lib/scan/scheduler.ts`) — held as a `Symbol.for("gitdash.scheduler")` global so HMR doesn't multiply it
3. Starts the **chokidar watcher** (`lib/scan/watcher.ts`)

Anything that needs DB/state must `await bootstrap()` before reading.

### Scheduler — three loops

- **Discovery** (every 10 min): `discoverRepos` walks roots up to `maxDepth`, finds `.git` children, marks stale repos as deleted via `markRepoDeleted`. Excludes `node_modules` and the long list of cache/system dirs in `DEFAULT_CONFIG`.
- **Local snapshot** (on demand + post-action): `collectSnapshot` runs `git status --porcelain=v2 --branch` plus `git log -1` and writes via `upsertSnapshot`. Concurrency-limited via `p-limit` (default 8).
- **Remote tick** (staggered round-robin, target `remoteIntervalMs`/repo): `compareWithRemote` calls `gh api repos/:owner/:name/commits/:branch` then `compare/:local...:remote`, both ETag-cached in the `gh_etag_cache` table. Concurrency limit 4. Returns `{state, ahead, behind}` written into the snapshot.

After every write the scheduler emits via the **store** (`lib/state/store.ts`), which is a singleton EventEmitter. The SSE route at `app/api/stream/route.ts` subscribes and re-pushes snapshots to connected browsers.

### Live UI

`Dashboard.tsx` opens an `EventSource` to `/api/stream`. Three event types:
- `snapshot` — full initial dump
- `bulk` — full re-dump (after discovery cycle)
- `update` — single-repo nudge; client refetches `/api/repos` to get fresh data

`deriveState` in `store.ts` is the single source of truth for what bucket each repo lands in (`clean | ahead | behind | diverged | dirty | weird | no-upstream | unknown`). The `Dashboard` then maps those to display `GroupKind`s — keep these in sync.

### Action pipeline

User clicks a row button → opens `ActionModal` → `POST /api/repos/[id]/actions/[name]` (with `x-csrf-token` header). The route validates CSRF + the action name against the whitelist in `lib/git/actions.ts::VALID_ACTIONS`, then calls `startAction()`, which:
1. Inserts an `actions_log` row
2. Spawns the right binary (git, editor, terminal) with appropriate flags
3. Returns `{runId}`

The modal then opens a second SSE on `/api/actions/[runId]/stream` that emits `line` events for each stdout/stderr line and a final `done` event with the exit code. After completion, the scheduler is asked to `collectOne(repoId)` so the UI updates.

**`commit-push` is special**: it's a multi-step action (`git add -A` → `git commit -m <msg>` → `git push`) implemented as `executeCommitPush` (Promise-based sequence). The route reads `commitMessage` from the JSON body and passes it through. The modal preview fetches `/api/repos/[id]/changes` first to show file count and flag suspicious files (`.env*`, `*.key`, `*.pem`, `id_rsa*`, `credentials*`, files >10MB).

### Adding a new action

1. Add to `ActionName` union, `buildArgs`, and `VALID_ACTIONS` in `lib/git/actions.ts`
2. If it's a launch-and-forget (like `open-editor`/`open-terminal`), branch on `isLaunch` in `executeRun` so it spawns detached and emits `done` immediately — the timeout will SIGTERM otherwise
3. If it needs args from the request (like `commit-push`'s message), extend `StartOptions` and parse the body in `app/api/repos/[id]/actions/[name]/route.ts`
4. Wire UI in `RepoCard.tsx` (`primaryAction` decides per-group buttons) or `RowMenu`

### Adding a snapshot field

Edit `lib/db/schema.ts` (add to `CREATE TABLE` — there's no migrations system, so adding a column to an existing DB will silently fail; document/instruct DB rebuild). Then update `SnapshotRow`/`SnapshotRaw`/`upsertSnapshot`/`rawToSnapshot` in `lib/db/repos.ts`, and consume in `RepoView`/`deriveState` (`store.ts`).

## Security model

- **Host header validation** (`middleware.ts` + `lib/security/host.ts`) — only loopback, RFC1918, link-local, and `100.64/10` (CGNAT/Tailscale) are allowed. Public-internet hosts get 403.
- **CSRF token** — `getCsrfToken()` reuses `GITDASH_CSRF_TOKEN` from env, otherwise generates one. All mutating routes call `validateCsrf` (constant-time compare).
- **No shell**: every `git` and `gh` invocation goes through `execFile`/`spawn` with `-C <repoPath> <args>`, never through `bash -c`. `assertSafeRef` regex-validates branch refs before they hit `git merge` etc.
- **Read-only/mutating allowlists** in `lib/git/exec.ts::runGit` — the function refuses git subcommands not on the allowlist for the requested mode.
- **Action whitelist** — only names in `VALID_ACTIONS` can be invoked from the API.

When extending: keep these guards. Don't reach for `bash -c`, don't bypass the allowlists.

## Auto-heal contract — NEVER show "Application error: a client-side exception"

That bare Next.js error page is a North Star violation: a beginner stuck on it has nothing to click. Four layers prevent it from ever being seen, and you must keep all four working when you change client code:

1. **`app/global-error.tsx`** — root error boundary. Replaces Next's default. Detects `ChunkLoadError` (or any chunk-load pattern) via `lib/autoheal.ts::isChunkLoadError` and hard-reloads immediately. For other errors, shows a friendly UI and auto-reloads after 5s.
2. **`app/error.tsx`** — page-level boundary. Same contract, with `reset()` exposed as a soft retry. Catches errors thrown inside the dashboard tree.
3. **`components/ChunkErrorGuard.tsx`** — mounted once in `app/layout.tsx`. Subscribes to `window.error` and `window.unhandledrejection` to catch async chunk failures (lazy `import()`, dynamic CSS) that bypass React boundaries because they happen outside the render cycle.
4. **`components/UpdateBanner.tsx` + `next.config.mjs::generateBuildId`** — `BUILD_ID` is pinned to `<commit>-<builtAt>`. The banner polls `/api/version` every 30s; on commit mismatch it auto-`hardReload`s after 5s, so a stale tab can't keep pointing at chunk hashes the new build deleted.

All four call into `lib/autoheal.ts`, which provides:

- `isChunkLoadError(err)` — pattern-matches the known ChunkLoadError shapes.
- `tryConsumeReloadBudget()` — sessionStorage-backed reload-loop guard. Caps auto-reloads at 3 inside a 30s window so a truly broken build can't spin the browser. After the budget is exhausted, the boundaries fall back to the friendly UI ("Auto-reload paused — click Reload to try once more").
- `hardReload()` — `window.location.replace` with a cache-busting `_gitdash_heal=<ts>` query so proxies / disk cache can't keep handing back the stale HTML.

**When adding new client features:**
- Don't catch errors silently in a way that hides them from the boundaries — let them bubble so the auto-heal can run.
- If you add a new lazy `import()` or dynamic chunk, you don't need to do anything special; `ChunkErrorGuard` already covers it.
- If you bump the version-poll interval or move `/api/version`, update `UpdateBanner` and re-test the auto-reload path.
- `lib/autoheal.ts` must remain client-safe (no `node:` imports). It's imported by both server-rendered (`global-error.tsx`) and client (`UpdateBanner`, boundaries) code.

**How to verify locally** (do this before merging anything that touches the auto-heal layer):
1. Build, start, open the dashboard.
2. Throw a synthetic error from a component (`throw new Error('boom')`) → `app/error.tsx` should render with auto-reload countdown.
3. Throw `new Error('ChunkLoadError: Loading chunk 5 failed')` from a component → page should hard-reload within ~1s.
4. Trigger the reload-loop guard (throw on every render) → after 3 reloads in 30s the friendly UI sticks instead of looping.

## Things that bite

- The scheduler/watcher/store singletons live on `globalThis` via `Symbol.for(...)` keys to survive Next.js HMR. Don't `new Scheduler()` directly anywhere else.
- `npm run dev` and `bin/gitdash dev` will both spawn discovery walks at startup — first boot can take a minute as remote comparisons stagger in.
- `next start` (and `bin/gitdash start`) loads the build artifact ONCE at boot. After `npm run build`, the running process must be restarted to pick up changes — there's no hot reload. Killing the listener PID and re-running `bin/gitdash start` is the supported flow.
- `gh` ETag cache lives in the SQLite DB. If you wipe the DB, expect a brief burst of full GitHub API calls until ETags re-populate.
- Repos with `.git` as a *file* (worktrees) are handled in `detectWeirdFlags` — preserve that path resolution if you touch it.
- **`JSON.parse` on external data is always wrapped in try/catch.** Includes SSE event payloads, `fetch().then(r => r.json())`, `localStorage`, and any `postMessage` handler. A single malformed frame must not throw out of an event listener. The Dashboard SSE handlers (`components/Dashboard.tsx:192-205`) are the canonical pattern: try/catch + `console.error` + drop the frame. Without this guard the auto-heal boundary catches the throw, but you eat a full reload for what could have been a one-frame skip.
- **`productionBrowserSourceMaps: true` stays on in `next.config.mjs`.** Self-hosted, LAN-only tool, no public attack surface. Debugging a beginner's "it crashed" report on a minified stack is impossible. The 20% bundle-size hit is worth it. Don't turn it off "to ship faster".
